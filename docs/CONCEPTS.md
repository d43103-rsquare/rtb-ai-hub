## 1. RTB AI Hub란 무엇인가?

### 한 문장 요약

**"팀원들 사이에서 업무를 조율하고, 반복 작업을 자동화하며, 회사 지식을 즉시 제공하는 AI 비서 시스템"**

### 비유로 이해하기

```
전통적인 협업                    RTB AI Hub 도입 후
─────────────────────           ─────────────────────
    👤 PM                           👤 PM
     │                               │
     ├─────┐                         ↓
     ↓     ↓                      [🤖 AI Hub]
    👤 기획  👤 디자이너               │
     │     │                    ┌────┼────┐
     ↓     ↓                    ↓    ↓    ↓
    👤 개발자                   👤 기획 디자이너 개발자
                               (맥락 자동 전달)
                               (회사 용어 자동 번역)
                               (진행 상황 자동 요약)
```

**기존 방식의 문제:**

- PM → 기획자: "로그인 기능 만들어주세요"
- 기획자 → 개발자: "PM이 뭐라고 했더라...?" (맥락 유실)
- 개발자: "이 테이블명이 뭐였지...?" (회사 지식 검색에 시간 소비)

**RTB AI Hub 방식:**

- PM → AI Hub → 기획자: "로그인 기능 (RTB-Wiki: 인증 프로세스 A, 테이블: usr_auth_mst)"
- AI Hub가 중간에서 **맥락 보존 + 회사 용어 자동 주입 + 적절한 담당자에게 전달**

## 2. 왜 필요한가? (문제 정의)

### 현실의 고통 포인트 🔥

#### 2.1 커뮤니케이션 단절

```
문제 상황:
- 디자이너: "사용자 프로필 페이지 완성했어요!"
- 개발자: "이 디자인의 '사용자'가 어떤 DB 테이블을 의미하나요?"
- 디자이너: "그건... 기획서를 봐야 할 것 같은데..."
```

**왜 일어나나?**

- 각 직군마다 다른 언어 사용 (디자이너: UX 용어, 개발자: 기술 용어)
- 정보가 Figma, Jira, Slack, Wiki 등에 분산
- 맥락 전달 실패 → 재작업 → 일정 지연

#### 2.2 반복 작업의 늪

```
매주 반복되는 작업:
- 월: Figma 디자인 → Jira 티켓 생성 (수동)
- 화: 개발 착수 → 회사 Wiki 검색 (20분)
- 수: PR 생성 → 리뷰어에게 맥락 설명 (30분)
- 목: QA 핸드오프 → 다시 맥락 설명 (20분)
- 금: 배포 → 모니터링 체크리스트 확인 (15분)
```

**한 주에 85분 = 연간 약 74시간 (9일)을 반복 작업에 소비**

#### 2.3 신입 사원의 고군분투

```
신입 개발자 첫날:
- "obj_bld_mst 테이블이 뭐죠?"
- "prd_pdm_mst는요?"
- "이 API 어디서 호출하나요?"

→ Wiki 검색 → 선배에게 질문 → 다시 Wiki 검색 → 하루 종일 온보딩...
```

**결과:**

- 생산적인 기여까지 2-3주 소요
- 반복적인 질문으로 팀 생산성 저하
- 온보딩 담당자의 업무 부담

## 3. 어떻게 해결하나? (솔루션 개념)

### 3.1 핵심 아이디어: "AI가 팀의 중간자 역할"

```
전통적인 Linear 워크플로우:
PM → 기획 → 디자인 → 개발 → QA → 배포
(각 단계마다 맥락 유실)

RTB AI Hub 방식:
        [🤖 AI Communication Coordinator]
          ↗️      ↓      ↓      ↓      ↘️
        PM   기획  디자인  개발  QA  배포
        (모든 단계의 맥락을 AI가 중앙 관리)
```

### 3.2 3대 핵심 기능

#### ① Communication Coordinator (커뮤니케이션 조율자)

- **역할**: 각 직군 사이의 통역사
- **방법**: 같은 정보를 역할별로 다르게 표현
- **예시**:

  ```
  원본 정보: "사용자 인증 기능 추가"

  → PM에게: "비즈니스 가치: 보안 강화, 예상 공수: 3일"
  → 개발자에게: "테이블: usr_auth_mst, API: POST /api/auth/login"
  → QA에게: "테스트 시나리오: 로그인 성공/실패 케이스"
  ```

#### ② 7-Agent Digital Twin (디지털 트윈 에이전트)

- **역할**: 각 직무자를 대신하는 AI 에이전트
- **구성**:
  - 🤖 PM Agent (VisionKeeper) - 비즈니스 우선순위 조율
  - 🤖 System Planner Agent (BlueprintMaster) - 아키텍처 설계
  - 🤖 UX Designer Agent (ExperienceCraftsman) - 사용자 경험
  - 🤖 UI Developer Agent (PixelPerfect) - 프론트엔드
  - 🤖 Backend Developer Agent (DataGuardian) - 백엔드
  - 🤖 QA Agent (QualityGatekeeper) - 품질 보증
  - 🤖 Ops Agent (InfrastructureKeeper) - 인프라 운영

- **동작 방식**:

  ```
  Slack에서 명령:
  "로그인 기능 만들어줘 (RNR-123)"

  → 7개 Agent가 자동으로 협업:
    - PM Agent: 요구사항 분석
    - System Planner: DB 스키마 확인
    - UX Designer: 사용자 플로우 검토
    - Backend Dev: API 코드 생성
    - QA: 테스트 케이스 생성

  → 결과물:
    - Jira 티켓 업데이트
    - GitHub PR 생성
    - Slack 알림 (각 직군에 맞춤 메시지)
  ```

#### ③ Active Wiki Knowledge (능동적 지식 제공)

- **역할**: 회사 지식을 AI가 자동으로 찾아서 주입
- **방법**:
  1. Jira 티켓에서 키워드 자동 추출
  2. RTB-Wiki에서 관련 문서 검색
  3. 개발 프롬프트에 자동 삽입
- **예시**:

  ```
  Jira 티켓: "빌딩 정보 조회 API 개발"

  AI 자동 처리:
  1. "빌딩" 키워드 감지
  2. RTB-Wiki에서 "obj_bld_mst 테이블" 문서 찾음
  3. 개발 시 자동으로 정확한 테이블명 사용

  결과: 개발자가 Wiki 검색할 필요 없음!
  ```

## 4. 핵심 개념 7가지

### 개념 1: Communication Pattern (커뮤니케이션 패턴)

**"같은 말도 듣는 사람에 따라 다르게 표현하기"**

```
원본: "로그인 기능 개발 완료"

→ PM:
  ✅ 기능: 로그인
  📊 진행률: 100%
  ⏱️ 예정: 금요일 배포

→ QA:
  ⚠️ 테스트 필요: 로그인 성공/실패 케이스
  📋 체크리스트:
     - [ ] 정상 로그인
     - [ ] 잘못된 비밀번호
     - [ ] 계정 잠금

→ 개발자:
  🔧 PR: #42 머지 완료
  🗄️ DB 마이그레이션: usr_auth_mst 테이블 추가
  📦 배포 영향: 인증 서비스 재시작 필요
```

### 개념 2: Role Adapter (역할 변환기)

**"각 직군의 언어를 서로 번역해주는 통역사"**

| 입력 (PM 언어)  | Role Adapter | 출력 (개발자 언어)    |
| --------------- | ------------ | --------------------- |
| "사용자 로그인" | 🔄           | `usr_auth_mst` 테이블 |
| "빌딩 정보"     | 🔄           | `obj_bld_mst` 테이블  |
| "매물 상세"     | 🔄           | `prd_pdm_mst` 테이블  |

**실제 동작:**

```javascript
// PM이 Jira에 작성:
"사용자가 빌딩 목록을 볼 수 있어야 함"

// Role Adapter 처리 후 개발자가 받는 내용:
"사용자가 빌딩 목록을 볼 수 있어야 함

📚 RTB-Wiki 참고:
- 테이블: obj_bld_mst (빌딩 마스터)
- 주요 컬럼: bld_id, bld_nm, addr
- 관련 API: GET /api/buildings
- 예제 쿼리: SELECT * FROM obj_bld_mst WHERE..."
```

### 개념 3: Context Engine (맥락 엔진)

**"누가, 언제, 왜 했는지 모든 이력을 연결"**

```
Jira 티켓 RNR-123 "로그인 기능"
  │
  ├─ Figma: 로그인 화면 디자인 링크
  ├─ GitHub PR #42: 구현 코드
  ├─ Preview: https://preview.rtb.dev/login
  ├─ 배포: 2026-02-12 15:30 (Production)
  └─ Decision: "JWT 방식 선택 (이유: ...)

모든 정보가 하나로 연결!
```

**활용:**

- QA가 RNR-123만 보면 디자인부터 배포까지 전체 맥락 파악
- 신입이 과거 의사결정 이유를 즉시 확인
- PM이 기능별 전체 히스토리를 한눈에 조회

### 개념 4: Smart Handoff (스마트 핸드오프)

**"업무 전달 시 자동으로 브리핑 자료 생성"**

```
상황: 디자인 완료 → 개발 시작

기존 방식:
PM: "디자이너님, 개발팀에게 전달 부탁드립니다"
디자이너: "개발자님, Figma 링크 보내드립니다"
개발자: "이게 어떤 기능이죠?" (30분 회의 소집...)

RTB AI Hub:
Jira 상태: "디자인 완료" → "개발 중"
      ↓
AI가 자동 브리핑 생성:
━━━━━━━━━━━━━━━━━━━━━━━━━
📋 개발팀에게 전달사항
━━━━━━━━━━━━━━━━━━━━━━━━━
✅ 디자인: Figma 링크
📝 요구사항: 로그인 + 비밀번호 재설정
🗄️ 관련 테이블: usr_auth_mst, usr_pwd_hist
⚠️ 주의사항: 기존 세션 로직과 호환 필요
📚 참고: RTB-Wiki 인증 프로세스 문서
━━━━━━━━━━━━━━━━━━━━━━━━━

→ Slack으로 자동 전송
→ 개발자는 브리핑만 보고 즉시 착수!
```

### 개념 5: Blocker Detection (블로커 감지)

**"일이 막혔는지 AI가 자동으로 감지하고 알림"**

```
감지 시나리오:
- Jira 티켓이 "In Progress"인데 3일간 업데이트 없음
- PR이 "리뷰 대기" 상태로 24시간 이상 방치
- QA 테스트가 "블로킹" 상태로 2일 경과

AI 자동 처리:
[⚠️ Blocker Alert - RNR-123]

티켓: 로그인 기능 개발
상태: In Progress (3일간 변경 없음)
담당자: @김개발
최근 활동: 2026-02-09 15:30

💡 제안:
- PR이 아직 생성되지 않았습니다
- 도움이 필요하신가요?
- 일정 조정이 필요한가요?

→ Slack DM + 리드에게 알림
```

### 개념 6: Decision Journal (의사결정 저널)

━━━━━━━━━━━━━━━━━━━━━━━━━
날짜: 2026-02-12
티켓: RNR-123
결정자: @김개발
태그: #인증 #아키텍처 #보안

결정 내용:
- JWT 토큰 기반 인증 채택
- 세션 방식 대신 선택

이유:
✅ 확장성 향상
✅ Redis 부하 감소
✅ MSA 환경에 적합

영향:
- 기존 세션 로직 제거
- 토큰 갱신 로직 추가

→ Decision DB에 자동 저장
→ 주간 다이제스트에 포함
→ Wiki 검색 가능
```

## 5. 실제 사용 시나리오

### 시나리오 1: 신규 기능 개발 (로그인 페이지)

```
1️⃣ PM이 Slack에서 시작:
   /rtb-ai start RNR-123

2️⃣ AI가 자동으로:
   - PM Agent: Jira 티켓 분석
   - System Planner: 필요한 테이블 확인 (usr_auth_mst)
   - UX Designer: Figma 디자인 검토
   - Backend Dev: API 코드 생성

3️⃣ 30분 후 결과:
   ✅ GitHub PR 생성 (#42)
   ✅ 테스트 코드 포함
   ✅ PR Description에 전체 맥락 자동 첨부
   ✅ 리뷰어에게 Slack 알림 (맥락 포함)

4️⃣ 리뷰 완료 후:
   ✅ QA Agent가 테스트 케이스 자동 생성
   ✅ QA에게 스마트 핸드오프 (브리핑 자동 전달)

5️⃣ QA 완료 후:
   ✅ Ops Agent가 배포 체크리스트 생성
   ✅ 배포 후 모니터링 자동 시작

총 소요 시간: 2시간 (기존 8시간)
```

### 시나리오 2: 장애 대응

```
1️⃣ 운영 서버 장애 발생:
   Datadog P1 Alert 발생

2️⃣ AI가 자동으로:
   - 로그/메트릭 분석
   - 근본 원인 파악
   - Jira 장애 티켓 자동 생성
   - 당직자에게 Slack 알림

## 8. FAQ (자주 묻는 질문)

### Q1: AI가 실수하면 어떻게 하나요?

**A:**

- AI는 **제안**만 하고, 사람이 **최종 승인**합니다.
- 모든 자동화는 **opt-in** 방식 (원하는 것만 활성화)
- 언제든지 **수동 오버라이드** 가능

### Q2: 기존 도구를 바꿔야 하나요?

**A:**

- 아니요! 기존 Jira, Slack, GitHub 그대로 사용
- RTB AI Hub는 **중간에서 연결**만 해줍니다
- 점진적으로 기능 추가 가능

### Q3: 개발자만 사용할 수 있나요?

**A:**

- 아니요! PM, 디자이너, QA 모두 사용 가능
- 각 역할에 맞는 인터페이스 제공
- 코딩 지식 불필요

### Q4: 비용은 얼마나 드나요?

**A:**

- AI API 사용량에 따라 과금 (월 약 $200-500)
- 절감되는 인력 시간을 고려하면 **ROI 10배 이상**

### Q5: 도입 기간은 얼마나 걸리나요?

**A:**

- 설치: 1일
- 파일럿: 2주
- 전사 확대: 4주
- **총 6주면 안정화 완료**
